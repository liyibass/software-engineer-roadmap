# [sre-8-6] 🔧 動手做：一次小型混沌實驗

> **本章目標**：照混沌工程的嚴謹流程，對一個服務做一次安全、可控的混沌實驗——砍掉一個元件，驗證系統會不會自己復原。

## 你會學到

- 把混沌工程的四步驟付諸實作
- 安全地注入故障（控制爆炸半徑）
- 用監控觀察系統的反應
- 從結果判斷韌性是否有效、該怎麼改善

## 概念說明

### 這一章在做什麼

Part 8-5 講了混沌工程的觀念，這一章你要**實際做一次**——選一個你前面部署的服務，砍掉它的一個元件，看系統會不會像你設計的那樣自己撐住或復原。

我們用一個最經典、最安全的實驗：**「砍掉一個容器，驗證它會自動重啟 / 流量會自動切換」**——這正好驗證你 infra Part 4-2（`Restart=always`）、Part 9-1（負載平衡跳過壞掉的後端）、SRE Part 8-3（故障轉移）學的東西。

```mermaid
graph LR
    A["① 定義穩定狀態<br/>看監控確認正常"]
    B["② 提出假設<br/>『砍一個容器，服務應仍正常』"]
    C["③ 注入故障<br/>docker kill 一個容器"]
    D["④ 觀察 + 學習"]
    A --> B --> C --> D
```

> ⚠️ 在**測試環境 / 你的 WSL 練習環境**做，不要在正式環境。控制爆炸半徑（Part 8-5）。

## 程式碼範例

### 前置：準備一個有冗餘的目標

要驗證「砍一個還撐得住」，目標得有冗餘（不然砍了當然掛）。用 infra Part 9-1 的負載平衡設定——一個 Nginx 後面掛多個 app 容器：

```bash
# 假設你有 Part 9-1 的 docker compose，跑了多個 app 副本 + nginx 負載平衡
docker compose up -d --scale app=3   # 跑 3 個 app 容器
docker compose ps                    # 確認 3 個 app + nginx 都在跑
```

---

### 第一步：定義穩定狀態（Part 8-5 ①）

先確認「正常時長什麼樣」。開一個持續打請求的視窗，建立基準：

```bash
# 持續每秒打一次，觀察正常時的回應
while true; do
  curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" http://localhost
  sleep 1
done
```

正常時應該一直看到 `200`、延遲穩定。**同時開著你的 Grafana 監控（Part 3-6）**，記下正常的錯誤率、延遲。

---

### 第二步：提出假設（Part 8-5 ②）

明確寫下你要驗證什麼：

```
假設：「如果我砍掉 3 個 app 容器中的 1 個，
       Nginx 應該會自動把流量導到剩下 2 個健康的容器，
       使用者頂多遇到極短暫的波動，整體服務維持正常（持續 200）。」
```

這驗證的是 infra Part 9-1 的「Nginx 自動跳過壞掉的後端」。

---

### 第三步：注入故障（Part 8-5 ③，小範圍）

**只砍一個**容器（控制爆炸半徑）。先找出容器名，然後砍：

```bash
docker compose ps              # 看 app 容器的名字
docker kill <某個app容器名>     # 故意砍掉「一個」
```

`docker kill` 模擬「一台機器突然掛掉」。注意——只砍一個，留著其他的，這就是「控制爆炸半徑」。

---

### 第四步：觀察並學習（Part 8-5 ④）

**立刻看你第一步那個持續打請求的視窗**，以及 Grafana 監控：

```
情況 A（假設成立 ✅）：
  請求視窗：可能閃過一兩個錯誤，然後馬上恢復一直 200
  → Nginx 偵測到那個容器掛了、自動跳過它，流量導到健康的 2 個
  → 韌性有效！你「證明」了負載平衡的故障處理真的管用

情況 B（假設不成立 ❌）：
  請求視窗：持續出現錯誤、或延遲飆高很久
  → 代表故障處理沒設好（例如健康檢查太慢、或沒設）
  → 太好了，你在「自己選的時間、測試環境、只砍一個」的可控情況下
    發現了這個弱點 → 可以從容修（調整 Nginx 健康檢查設定等）
```

不管 A 或 B，你都贏了（Part 8-5 的精神）。

---

### 第五步：進階——驗證自動重啟

如果你的容器設了 `restart: always`（infra Part 5-5），還可以順便驗證「掛掉的那個會不會自己回來」：

```bash
# 砍掉後過幾秒，再看一次
docker compose ps
# 那個被砍的容器，應該已經被自動重啟、狀態回到 running
```

這驗證了 infra Part 4-2/5-5 的「自動重啟」——掛掉的容器自己復活了。**你親眼看到系統的「自我修復」（SRE Part 6-2 的 L5）在運作。**

---

### 收尾：記錄發現

把這次實驗寫成簡單記錄（像迷你 postmortem，Part 5-4）：

```
混沌實驗記錄：
- 假設：砍 1/3 容器，服務應維持正常
- 結果：✅ 成立。閃過 1 個錯誤後恢復，2 秒內 Nginx 跳過壞容器
       被砍的容器在 5 秒後自動重啟回來
- 發現：故障轉移與自動重啟都有效
- 改善點：那「1 個錯誤」是否能透過調整健康檢查頻率消除？（下次實驗）
```

## 小練習

### 練習 1：做一次容器混沌實驗

照本章步驟，對一個有冗餘的服務做實驗：定義穩定狀態 → 提假設 → 砍一個容器 → 觀察。系統撐住了嗎？

---

### 練習 2：驗證自我修復

如果你的容器設了 `restart: always`，砍掉後觀察它會不會自動回來。記錄花了幾秒復活。

---

### 練習 3：設計下一個實驗

想一個「更進階」的混沌實驗（但仍安全），例如：

- 模擬某個容器「變慢」而不是「掛掉」（更接近真實的部分故障）
- 砍掉資料庫，驗證 Part 8-1 的斷路器/降級有沒有生效

寫出你的假設、要注入什麼故障、怎麼控制爆炸半徑、觀察什麼。

> 你完成了「為失敗而設計」的全套——從韌性模式（8-1、8-2）、冗餘（8-3）、安全發布（8-4），到主動驗證（8-5、8-6）。最後一個 Part 9，我們談「文化」——讓這一切在團隊中真正運作起來。
