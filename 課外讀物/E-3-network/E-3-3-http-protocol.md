# [E-3-3] HTTP 協定詳解

> **你會了解**：HTTP 的動詞（Methods）分別代表什麼語意、狀態碼（Status Codes）怎麼解讀、常用 Headers 是什麼意思，以及 HTTP/2 相較於 HTTP/1.1 解決了什麼問題。

---

## 背景 / 故事

你有沒有遇過這種事：

後端同事說「我把 API 改成 PATCH 了」，你點點頭，但心裡在想——不是已經有 PUT 了嗎？有什麼差別？

或是前端抓 API 時，偶爾看到 `422 Unprocessable Entity`，和 `400 Bad Request` 感覺很像，但到底差在哪？

HTTP 這套協定定義了一套完整的「語言」，讓瀏覽器和伺服器可以精確地溝通。每個動詞、每個數字，都有它設計時的語意。一旦你搞清楚這套語言，閱讀 API 文件和 debug 網路問題都會順很多。

---

## 核心概念

### HTTP 是什麼

HTTP 全名是 **HyperText Transfer Protocol（超文字傳輸協定）**，是網頁世界最核心的應用層協定。

說白了：它是瀏覽器和伺服器之間說話的規則書。規定了「請求長什麼樣子」、「回應長什麼樣子」、「用什麼動詞表達什麼意思」。

---

### HTTP Methods — 動詞決定你的意圖

HTTP 的動詞（Method）告訴伺服器「我要對這個資源做什麼」。

> 把伺服器想成一座圖書館，URL 就是書的位置，Method 就是你的動作。

#### 五個最常用的動詞

**GET — 我要借書看（只讀，不改變任何東西）**

```
GET /books/42
```

向伺服器請求某個資源。不帶任何資料過去，也不改變伺服器的狀態。就像去圖書館查資料，書還留在架上。

**POST — 我要新增一本書（建立新資源）**

```
POST /books
Body: { "title": "深入淺出 HTTP", "author": "..." }
```

向伺服器送出資料，通常用於建立新的資源。每次呼叫可能產生新的結果（叫兩次 POST 可能建立兩筆紀錄）。

**PUT — 我要把整本書換掉（完整替換）**

```
PUT /books/42
Body: { "title": "新標題", "author": "新作者", "year": 2024 }
```

把指定資源整個替換掉。注意：你要提供完整的資料，PUT 是「全換」，而不是「改一點」。

**PATCH — 我要在書上某頁貼便利貼（部分更新）**

```
PATCH /books/42
Body: { "title": "只改標題就好" }
```

只更新資源的部分欄位。比 PUT 更精準，適合「我只想改某個欄位，其他保留」的情境。

**DELETE — 把書從館藏中移除**

```
DELETE /books/42
```

刪除指定資源。

---

#### 安全性（Safe）vs 冪等性（Idempotent）

這兩個概念常常出現在 API 設計的討論裡：

| Method | 安全（不改狀態）| 冪等（多次=一次）| 適用情境 |
|--------|:-----------:|:----------:|------|
| GET    | ✅ | ✅ | 取得資料 |
| POST   | ❌ | ❌ | 建立資源（每次都產生新的） |
| PUT    | ❌ | ✅ | 完整替換資源 |
| PATCH  | ❌ | ❌（視實作） | 部分更新 |
| DELETE | ❌ | ✅ | 刪除（刪兩次和刪一次結果一樣） |

- **安全（Safe）**：呼叫這個方法，伺服器的狀態不會改變。GET 查詢不管幾次，資料都還在。
- **冪等（Idempotent）**：重複呼叫多次，效果和呼叫一次一樣。PUT 把書換成「新標題」，不管換幾次，結果都是「新標題」。

> 為什麼要在乎冪等性？因為網路不可靠，請求可能因為 timeout 被重送。如果你的 API 是冪等的，重送不會造成重複資料或奇怪的副作用。

### Status Codes — 伺服器的回覆語氣

HTTP 狀態碼是一個三位數，第一個數字代表「大類」：

```
2xx → 成功了！
3xx → 去別的地方找
4xx → 你的問題（客戶端錯誤）
5xx → 我的問題（伺服器錯誤）
```

> 用餐廳來想像：你是客人，服務生是伺服器。

#### 2xx — 成功上菜

| 狀態碼 | 名稱 | 意思 |
|--------|------|------|
| **200 OK** | 一切正常 | 最常見的成功回應，你點的餐成功送上來了 |
| **201 Created** | 建立成功 | 通常是 POST 之後回傳，你的訂單已建立 |
| **204 No Content** | 成功但沒有回傳內容 | DELETE 成功後常用，菜已收走，沒什麼好說的 |

#### 3xx — 去別的地方找

| 狀態碼 | 名稱 | 意思 |
|--------|------|------|
| **301 Moved Permanently** | 永久搬家 | 這道菜從今以後都在隔壁餐廳賣了 |
| **302 Found** | 暫時在別的地方 | 今天廚師請假，這道菜暫時去隔壁做 |

#### 4xx — 客人的問題

| 狀態碼 | 名稱 | 意思 |
|--------|------|------|
| **400 Bad Request** | 請求格式錯誤 | 你點餐的方式完全看不懂，能重點一次嗎？ |
| **401 Unauthorized** | 沒有認證 | 這是會員專屬餐廳，請先登入 |
| **403 Forbidden** | 有認證但沒權限 | 你有會員資格，但這桌是 VIP 專屬，你進不來 |
| **404 Not Found** | 找不到資源 | 菜單上根本沒有你點的這道菜 |
| **422 Unprocessable Entity** | 格式對但內容無法處理 | 你的訂單格式正確，但你要的份量是 `-5` 份，這沒辦法做 |

> **401 vs 403 的差別**
> 401 是「我不知道你是誰」，你需要提供身分證明（登入）。
> 403 是「我知道你是誰，但你沒有資格」，就算再次登入也沒用。

#### 5xx — 廚房的問題

| 狀態碼 | 名稱 | 意思 |
|--------|------|------|
| **500 Internal Server Error** | 伺服器內部錯誤 | 廚房失火了，不是你的錯 |
| **503 Service Unavailable** | 服務暫時無法使用 | 餐廳今天休息或爆滿，請稍後再試 |

### HTTP Headers — 夾在信封裡的便條紙

Headers 是 Request 和 Response 裡夾帶的「附加說明」，不算是主要內容，但對傳輸行為影響很大。

#### Request Headers（你帶過去的便條）

**`Content-Type`** — 告訴伺服器你帶的資料是什麼格式

```
Content-Type: application/json
Content-Type: multipart/form-data
```

如果你發送的是 JSON 但沒有告訴伺服器，它可能完全看不懂你的 body。

**`Authorization`** — 你的身分證明

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

後端 API 通常要求你在 Header 裡帶上 Token，才知道你是誰、有什麼權限。

**`Cookie`** — 瀏覽器自動附上的狀態

```
Cookie: session_id=abc123; user_pref=dark_mode
```

Cookie 是存在瀏覽器裡的一小塊資料，每次請求都會自動附上，讓伺服器知道「你上次在這裡」。

#### Response Headers（伺服器回給你的便條）

**`Content-Type`** — 告訴你回傳的資料是什麼格式

```
Content-Type: text/html; charset=utf-8
Content-Type: application/json
```

**`Cache-Control`** — 這份資料可以快取多久

```
Cache-Control: max-age=3600
Cache-Control: no-store
```

`max-age=3600` 代表這份資料可以被瀏覽器快取一小時，下次同樣的請求就不用再跑到伺服器了。`no-store` 代表完全不要快取（通常用在敏感資料）。

### HTTP/1.1 vs HTTP/2 — 從單線道到多車道

你有沒有觀察過，打開一個現代網站時，Network 面板裡同時有幾十個請求在跑？

在 HTTP/1.1 的時代，這其實是一個大問題。

#### HTTP/1.1 的瓶頸：排隊

HTTP/1.1 的每條 TCP 連線同一時間只能處理一個請求。前一個沒收到回應，後面的就要等。這個問題叫 **Head-of-Line Blocking（隊頭阻塞）**。

```
HTTP/1.1 的世界（單一連線）：

請求 A ──────── 回應 A
                          請求 B ──── 回應 B
                                               請求 C ── 回應 C
```

瀏覽器的解法是「多開幾條連線」（通常 6 條），但這只是治標，連線本身有成本。

#### HTTP/2：多路復用（Multiplexing）

HTTP/2 的核心改進是讓一條 TCP 連線可以同時處理多個請求，互不干擾。

```
HTTP/2 的世界（同一條連線）：

── 請求 A ── 請求 B ── 請求 C ──→
← 回應 B ── 回應 A ── 回應 C ──
（順序可以打亂，各自獨立）
```

> 把 HTTP/1.1 想成單線道省道：每輛車要等前面的車走完才能走。
> HTTP/2 是多車道高速公路：很多車同時跑，互不影響。

HTTP/2 還帶來了幾個額外好處：
- **Header 壓縮**：Headers 往往重複性很高，HTTP/2 會壓縮，減少傳輸量
- **Server Push**：伺服器可以主動推送資源，不等你請求（例如你才請求 HTML，伺服器順手把 CSS 也推過來）

現代瀏覽器和主流伺服器（Nginx、Apache、Node.js）都支援 HTTP/2，你不需要特別改程式碼，配好憑證（需要 HTTPS）就可以自動享受。

---

## 小結

HTTP 是一套精心設計的語言：動詞表達意圖，狀態碼傳遞結果，Headers 夾帶附加資訊。理解這套語言，讓你在設計 API 時做出更準確的語意決策，在 debug 時也能直接從回應碼判斷問題出在哪一側。HTTP/2 的多路復用解決了 HTTP/1.1 的排隊問題，讓現代網頁載入速度大幅提升。
