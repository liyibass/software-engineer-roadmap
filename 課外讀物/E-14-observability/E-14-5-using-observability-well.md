# [E-14-5]【深入版】怎麼把觀測性「用好」

> **目標**：超越工具，學會「把觀測性用好」的實務——該記什麼日誌、結構化日誌、關聯 ID，以及它和告警的關係。

## 工具不等於用得好

E-14-1~4 講了觀測性的概念與工具。但「**裝了 ELK、Grafana ≠ 觀測性做得好**」——很多團隊有一堆工具，出事時還是查不到原因。這篇講「**怎麼真正用好**」。

## ① 該記什麼日誌（與不該記什麼）

日誌不是「越多越好」——記錯東西，反而淹沒了重要資訊、又浪費儲存。

**該記的**：

- **關鍵事件**：請求進來/結束、重要的業務動作（下單、付款）、狀態轉換。
- **錯誤與例外**：完整的錯誤訊息、堆疊（stack trace）、發生時的上下文（哪個使用者、什麼參數）。
- **足以「重建現場」的上下文**：出事時，光看日誌能不能還原「發生了什麼」？

**不該記的**：

- **敏感資料**：密碼、信用卡號、個資——**絕對不能寫進日誌**（資安事故！呼應 E-10 安全）。日誌常被很多人看、被集中儲存，洩漏了很嚴重。
- **無意義的雜訊**：每個迴圈都印一行、debug 用的垃圾訊息留在正式環境——淹沒真正重要的日誌。

原則：**記「出事時你會想看的」，不記「平常的雜訊」和「敏感資料」。**

## ② 結構化日誌（Structured Logging）

這是「用好日誌」的關鍵技巧。對比兩種日誌：

```
// ❌ 非結構化（純文字）—— 難搜尋、難分析
"User 123 failed to login at 14:03 because wrong password"

// ✅ 結構化（JSON 等）—— 好搜尋、好分析
{ "event": "login_failed", "user_id": 123, "time": "14:03",
  "reason": "wrong_password", "ip": "1.2.3.4" }
```

**結構化日誌**把日誌寫成「**有欄位的格式（如 JSON）**」，而非純文字。好處：

- **好搜尋/過濾**：在 ELK/Kibana 能精準查「`event=login_failed` 且 `user_id=123`」——純文字只能模糊 grep。
- **好聚合/畫圖**：能統計「各 `reason` 的登入失敗次數」畫成圖。
- **機器好處理**：欄位明確，工具能自動解析。

現代後端框架都支援結構化日誌——**用它，別用純字串拼接的日誌**。這讓你的 ELK（E-14-3）威力大增。

## ③ 關聯 ID（Correlation ID / Trace ID）

這是分散式除錯的神器（呼應 E-14-3、sre Part 3-5 追蹤）。問題：一個請求穿過好幾個服務，每個服務各自記日誌——怎麼把「**同一個請求**」在所有服務的日誌串起來？

解法：**給每個請求一個唯一的「關聯 ID」，讓它貫穿所有服務的日誌。**

```
請求進來 → 產生一個 ID（如 "req-abc123"）
→ 這個 ID 隨請求傳到每個經手的服務
→ 每個服務記日誌時，都帶上這個 ID
→ 出事時，在 Kibana 搜尋 "req-abc123"
→ 瞬間撈出「這個請求在所有服務」的完整足跡 ✅
```

有了關聯 ID，你就能在集中式日誌（ELK）裡，**把散落各服務的日誌「串成一個請求的故事」**——這幾乎是分散式除錯的必備。它和分散式追蹤（Traces）是同源的概念。

## ④ 觀測性與告警的關係

觀測性提供「資料」，但「**出事時主動通知你**」要靠告警（sre Part 4）。兩者關係：

- **觀測性（資料）**：指標、日誌、追蹤——讓你「能查」。
- **告警（行動）**：基於這些資料，在「該採取行動時」通知你（sre Part 4）。

關鍵（呼應 sre Part 4 的智慧）：

- **對「症狀」告警**（錯誤率、延遲），用指標（Metrics）——別對一堆原因告警（告警疲勞，sre Part 4-2）。
- **日誌/追蹤是「告警響了之後，拿來查根因」的**——不是拿來告警的（日誌量太大，不適合直接觸發告警）。

所以：**指標 → 告警（發現問題）；日誌/追蹤 → 診斷（找原因）**。各司其職。

## ⑤ 一個務實的觀測性檢查清單

判斷你的觀測性「用得好不好」：

- [ ] 出事時，光靠手邊的資料（指標/日誌/追蹤），**能不能查到根因**？（這是終極檢驗）
- [ ] 日誌是**結構化**的嗎？（能精準搜尋/聚合）
- [ ] 有**關聯 ID** 串起跨服務的日誌嗎？
- [ ] 有沒有**不小心記了敏感資料**？
- [ ] 告警是對「症狀」、且「只在需要行動時響」嗎？（sre Part 4）
- [ ] 儀表板能讓人「10 秒看出健康狀況」嗎？（sre Part 3-4）

如果這些都做到，你的觀測性才算「用得好」——而不只是「裝了一堆工具」。

## 小結

- 工具 ≠ 用得好。要用好觀測性，重點在**怎麼用**。
- **該記什麼日誌**：關鍵事件、錯誤上下文；**別記**敏感資料、雜訊。
- **結構化日誌**（JSON）：好搜尋、好聚合，讓 ELK 威力大增。
- **關聯 ID**：串起跨服務的日誌，分散式除錯神器。
- **指標→告警（發現），日誌/追蹤→診斷（找因）**，各司其職。
