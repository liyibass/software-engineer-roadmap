# [E-5-1] 為什麼陣列從 0 開始？

> **你會了解**：陣列索引從 0 開始不是「大家說好的習慣」，而是有真實的記憶體運算原因——搞懂這件事，你對電腦的理解會深一個層次。

---

## 幾乎每個工程師都問過這個問題

你剛學程式，老師說：「陣列的第一個元素是 `arr[0]`，第二個是 `arr[1]`。」

你心裡 OS：「…為什麼不是 `arr[1]` 和 `arr[2]`？人類數數不都從 1 開始嗎？」

這不是個蠢問題。這是個好問題。而且它有一個真正的技術答案。

---

## 先理解記憶體長什麼樣子

電腦的記憶體（RAM）可以想像成一條超長的格子帶，每個格子都有一個編號，叫做**記憶體位址**（memory address）。

```
位址：  1000  1001  1002  1003  1004  1005  ...
       [ A  ][ B  ][ C  ][ D  ][ E  ][ F  ] ...
```

就像一棟公寓大樓，每個房間都有房號。房號不代表「第幾間」，它代表的是「這間房在哪裡」。

電腦存東西的時候，就是找一塊空的格子，把資料放進去，然後記住那個位址。

---

## 陣列在記憶體裡長什麼樣子

陣列有個關鍵特性：**元素在記憶體中是連續排列的**。

假設你有一個整數陣列 `[10, 20, 30]`，每個整數佔 4 bytes，陣列的起始位址（我們叫它 `base`）是 1000：

```
位址：  1000    1004    1008
       [ 10  ] [ 20  ] [ 30  ]
        [0]     [1]     [2]
```

你看出規律了嗎？

```
arr[0] 的位址 = base + 0 * 4 = 1000
arr[1] 的位址 = base + 1 * 4 = 1004
arr[2] 的位址 = base + 2 * 4 = 1008
```

用一個公式表示就是：

```
第 i 個元素的位址 = base + i * 每個元素的大小
```

這個公式簡潔、直接，CPU 執行一次乘法加一次加法就找到了。

---

## 如果從 1 開始會怎樣？

如果陣列索引從 1 開始，「第 i 個元素」的計算就變成：

```
第 i 個元素的位址 = base + (i - 1) * 每個元素的大小
```

多了一次減法。

```
arr[1] 的位址 = base + (1-1) * 4 = base + 0   → 1000
arr[2] 的位址 = base + (2-1) * 4 = base + 4   → 1004
arr[3] 的位址 = base + (3-1) * 4 = base + 8   → 1008
```

這個 `(i - 1)` 在現代 CPU 上只是一道指令，快得微不足道。但在電腦發展的早年，這一道多餘的指令是真的有代價的。更重要的是，這破壞了公式的對稱美感——索引 `i` 和偏移量 `i * size` 之間那個乾淨的對應關係消失了。

從 0 開始，索引本身就**等於**從 base 出發的「跳幾格」，不需要任何校正。

---

## Dijkstra 的一篇短文

1982 年，傳奇電腦科學家 **Edsger W. Dijkstra**——就是那個發明最短路徑演算法的人——寫了一篇不到一頁的手寫短文，標題叫《Why numbering should start at zero》（為什麼編號應該從零開始）。

他用數學的角度論證，表達一個範圍 0 到 n 的序列，最好的寫法是：

```
0 ≤ i < n
```

而不是：

```
1 ≤ i ≤ n
```

理由是：
- 下界用 `≤`，上界用 `<`，這樣範圍的大小剛好等於 `n - 0 = n`，算起來最自然
- 空範圍可以寫成 `0 ≤ i < 0`，不需要用負數表示下界
- 最重要的是：這個慣例和 **off-by-one error**（差一錯誤）有直接關係

---

## Off-by-one Error：程式界最常見的 bug

**Off-by-one error** 是指「差了一格」的錯誤，它長這樣：

```javascript
const fruits = ["蘋果", "香蕉", "芒果"];

// 錯誤版本：i <= arr.length 會越界！
for (let i = 0; i <= fruits.length; i++) {
  console.log(fruits[i]); // 最後一次：fruits[3] → undefined
}

// 正確版本
for (let i = 0; i < fruits.length; i++) {
  console.log(fruits[i]); // 0, 1, 2 → 剛好結束
}
```

`fruits.length` 是 3，但最後一個元素的索引是 2。

正是因為 0-indexed 搭配「`i < length`」這個慣例，讓迴圈的終止條件變得直觀——你不需要思考「要不要減一」，只要記住「`<` 就對了」。

如果從 1 開始，終止條件就會變成 `i <= length`，看起來沒問題，但一不小心就寫成 `i < length` 少跑了一次，或者邊界判斷混亂。

---

## 有哪些語言偏偏從 1 開始？

不是所有語言都選擇了 0-indexed。主要是幾個有**數學或科學計算背景**的語言：

| 語言 | 索引起點 | 背景 |
|------|---------|------|
| Lua | 1 | 嵌入式腳本語言，設計給非程式設計師 |
| MATLAB | 1 | 矩陣運算，數學家的語言 |
| R | 1 | 統計分析，統計學家的語言 |
| Fortran | 1 | 科學計算，1950 年代的語言 |

這些語言的設計者選擇了「對人類直覺更友善」的做法，代價是讓陣列存取的底層計算多了一個偏移量。

但現實的代價是：MATLAB 用戶轉 Python 的第一週，幾乎無一例外都會踩一次索引的坑。在 MATLAB 裡 `arr(1)` 是第一個元素，到了 Python 變成 `arr[0]`，肌肉記憶是很難改的。

---

## 小結

- 陣列從 0 開始，是因為「索引 = 從起點跳幾格」，公式最乾淨
- 如果從 1 開始，每次存取都要多一個 `(i-1)` 的偏移校正
- Dijkstra 從數學角度論證 0-indexed + `[0, n)` 這個慣例最不容易出錯
- Off-by-one error 是實際開發中高頻出現的 bug，理解 0-indexed 的邏輯能幫你少踩坑
- Lua、MATLAB、R 選擇從 1 開始，有其設計背景，但在多語言環境下容易造成混亂
