# [E-5-5] 為什麼工程師喜歡用 foo 和 bar 當變數名稱

> **目標**：輕鬆認識 `foo`、`bar` 這些「占位變數名」的由來，以及為什麼教學和範例裡到處都是它們。

## 到處都是的 foo 和 bar

看程式教學、文件、範例，你會一直看到這種變數名：

```
function foo(bar) {
    let baz = bar + 1;
    return baz;
}
```

`foo`、`bar`、`baz`——這些奇怪的字是什麼意思？它們其實**沒有意義**，是工程師慣用的「**占位符（placeholder）**」——當「名字不重要、只是舉例」時用的「隨便取的名字」。

## 為什麼用「沒意義」的名字

這其實是**刻意的**。當你在教一個「概念」（例如「函式怎麼傳參數」），如果用 `calculateUserDiscount(price)` 這種「有意義」的名字，學習者可能會分心去想「折扣怎麼算」——但你想教的只是「函式語法」本身。

用 `foo(bar)` 這種**沒意義的名字**，反而能讓人**專注在「結構/語法」上，而不被「業務含義」干擾**。它在說：「這裡放什麼名字不重要，重點是這個結構。」

> ⚠️ 但注意——`foo`/`bar` **只該用在「教學範例」**！在**真實程式碼**裡用 `foo`、`data`、`temp` 這種無意義的名字是**壞習慣**（違反 Clean Code 命名原則，E-6-2）。真實程式碼的變數名要「說明它是什麼」。這套課程的 CLAUDE.md 規範也特別禁止真實程式碼用 `foo`/`data`/`temp`。

## 它們的由來

`foo` 和 `bar` 的確切起源有點模糊，但有兩個主要說法：

**① 來自軍事俚語「FUBAR」**

二戰時期的軍事俚語 **FUBAR**（"Fouled Up Beyond All Recognition"，搞砸到認不出來）——工程師借用了它，拆成 `foo` 和 `bar`。

**② 來自更早的 "foo"**

`foo` 這個無意義的字，在更早的漫畫、塗鴉文化裡就出現過（1930 年代的漫畫就有 "foo"）。

不管確切起源，它們在 1960~70 年代的早期工程文化（尤其 MIT 等地）就被廣泛用作「占位名」，並隨著教科書、文件流傳至今。

## 一整套占位符

`foo`、`bar` 之後，還有一整套習慣的占位名，依序常是：

```
foo, bar, baz, qux, quux, corge, ...
```

需要更多占位名時就往後拿。你還會看到其他常見的占位：

- 占位人名：`Alice`、`Bob`（尤其密碼學/安全的例子，E-10）——「Alice 想傳訊息給 Bob」。
- 占位網域：`example.com`（這是官方保留給「範例」用的網域，不會真的被註冊）。

## 它反映的文化

`foo`/`bar` 和「bug 的飛蛾」（E-5-3）、「418 茶壺」（E-3-7）一樣，是工程文化的小傳統——**一套大家心照不宣的「共同語言」**。當你看到 `foo(bar)`，立刻知道「喔，這是個不用在意名字的範例」。這些小慣例，是工程師社群文化的一部分。

## 小結

- `foo`、`bar`、`baz` 是「無意義的占位變數名」，用在「教學範例」——讓人專注在結構/語法，不被業務含義干擾。
- **只該用在範例**！真實程式碼用無意義的名字是壞習慣（違反 Clean Code，E-6-2）。
- 由來可能來自軍事俚語 FUBAR 或更早的漫畫文化，流傳於早期工程界。
- 整套占位：foo/bar/baz/qux…；人名占位 Alice/Bob；網域占位 example.com。
