# [E-6-5] 注解的正確使用：解釋「為什麼」而非「做什麼」

> **目標**：理解好注解與壞注解的差別——好注解解釋「為什麼」，壞注解只是重複程式碼已經說的「做什麼」。

## 注解越多越好嗎？

新手常以為「注解越多 = 程式碼越好」。**錯。** 大部分的注解其實是**多餘、甚至有害**的。好的程式碼，注解通常很少——因為**程式碼本身就說清楚了「做什麼」**。

核心原則（這是這篇的精髓）：

> **好注解解釋「為什麼（why）」，而不是「做什麼（what）」。** 「做什麼」應該由「**程式碼本身 + 好的命名**」說清楚（E-6-2）。

## 壞注解：重複程式碼

最常見的壞注解，是「**把程式碼用中文再講一遍**」——毫無資訊量：

```javascript
// 壞注解：講了程式碼已經說的事
i = i + 1;          // 把 i 加 1   ← 廢話，程式碼已經說了

// 對陣列排序
array.sort();        // ← 廢話
```

這種注解：

- **沒有新資訊**：程式碼已經寫了 `i + 1`，注解「把 i 加 1」毫無幫助。
- **會過時、會說謊**：程式碼改了、注解忘了改 → 注解變成「**錯誤的資訊**」，比沒注解更糟（誤導讀者）。

## 好注解：解釋「為什麼」

好注解提供「**程式碼本身看不出來的『原因、背景、意圖』**」：

```javascript
// 好注解：解釋「為什麼」
// 必須先排序，因為下面的二分搜尋要求陣列有序
array.sort();

// 這裡用 setTimeout 0 是為了等瀏覽器完成重繪，
// 否則會讀到舊的尺寸（一個已知的瀏覽器 quirk）
setTimeout(measure, 0);

// 業務規定：VIP 客戶免運費門檻是 0（一般客戶是 1000）
const freeShippingThreshold = isVip ? 0 : 1000;
```

這些注解有價值，因為它們解釋了「**為什麼這樣寫**」——這是程式碼本身無法表達的（程式碼能說「做了什麼」，但說不出「為什麼這樣決定」）。

## 好注解的幾種類型

| 類型 | 例子 |
|------|------|
| **解釋「為什麼」** | 「先排序，因為二分搜尋需要有序」 |
| **警告後果** | 「改這個值會影響計費，務必先跟業務確認」 |
| **解釋「不直觀的決定」** | 「這裡故意不用快取，因為資料必須即時」 |
| **標記待辦** | `// TODO: 之後改成批次處理`、`// FIXME: 暫時 workaround` |
| **法律/版權** | 檔頭的授權聲明 |

## 最好的注解：不需要注解

一個進階觀念——**很多時候，與其寫注解，不如「改善程式碼讓它不需要注解」**：

```javascript
// 壞：用注解解釋一段難懂的條件
// 檢查使用者是否成年且已驗證
if (user.age >= 18 && user.verified && !user.banned) { ... }

// 更好：用「好命名的函式」取代注解（E-6-2、E-6-3）
if (canAccessContent(user)) { ... }
function canAccessContent(user) {
    return user.age >= 18 && user.verified && !user.banned;
}
```

第二種把「難懂的條件」抽成一個「會說話的函式名」`canAccessContent`——**程式碼自己就解釋了「做什麼」，連注解都不用了**。這就是 Clean Code 的精神：**讓程式碼自我解釋（self-documenting），而不是靠注解去補救難懂的程式碼**。

## 小結

- 注解不是越多越好——大部分「重複程式碼」的注解是多餘、甚至有害（會過時說謊）。
- **好注解解釋「為什麼」（原因、意圖、背景），不解釋「做什麼」**（那交給程式碼 + 好命名）。
- 好注解類型：解釋為什麼、警告後果、不直觀的決定、TODO/FIXME。
- 最好的注解是「**不需要注解**」——用好的命名和函式拆分，讓程式碼自我解釋。
