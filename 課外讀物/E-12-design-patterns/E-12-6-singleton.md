# [E-12-6] Singleton 模式：全域唯一的實例

> **目標**：理解 Singleton（單例）模式——確保某個東西「全程只有一個實例」，以及為什麼它好用、又為什麼常被批評。

## 一個概念：有些東西只該有一個

有些東西，整個程式裡「只該存在一個」：

- 資料庫連線池（建多個浪費資源）。
- 設定管理器（全程共用一份設定）。
- 日誌寫入器。

**Singleton 模式**就是確保「**一個類別，全程只有一個實例**，且提供一個全域存取點」。

用類比：一個國家只有一個「中央銀行」。不管誰要、從哪要，拿到的都是同一個中央銀行——不會有第二個。

## 怎麼做

核心做法：**把建構子藏起來（不讓外面隨便 new），改提供一個 `getInstance()`，它永遠回傳「同一個」實例**。

```
class 設定管理器:
    private static 實例 = null
    private 建構子() { ... }              // 藏起來，外面不能 new

    static function getInstance():
        如果 實例 == null:
            實例 = new 設定管理器()        // 第一次才建立
        return 實例                        // 永遠回傳同一個

// 使用：不管在哪呼叫，拿到的都是同一個
設定 = 設定管理器.getInstance()
```

第一次呼叫 `getInstance()` 建立實例，之後每次都回傳「那同一個」——全程唯一。

## 好處

- **節省資源**：像資料庫連線池，建一個共用就好。
- **全域一致**：所有地方共用同一份狀態（如設定）。
- **方便存取**：一個全域存取點，到處都能拿到。

## 為什麼 Singleton 常被批評

Singleton 是少數「**爭議很大**」的模式——很多人視它為「反模式」。原因：

**① 它是「偽裝的全域變數」**：全域可存取的共享狀態，容易被到處亂改、產生隱藏的依賴，讓程式難追蹤（呼應 E-6 反模式的「全域狀態」）。

**② 難測試**：因為它是全域的、又藏住建構子，測試時很難「換成假的」（不像依賴注入能輕鬆替換，呼應 E-9 測試替身）。它讓單元測試變難。

**③ 隱藏依賴**：一個函式裡用 `設定管理器.getInstance()`，從函式簽名看不出它依賴設定——依賴被藏起來了（違反「明確依賴」的原則）。

**④ 並發問題**：多執行緒下，`getInstance()` 若沒處理好，可能建出多個實例（要加鎖）。

## 現代的替代：依賴注入

現代開發更傾向用**依賴注入（Dependency Injection，DI）**來達成「共用單一實例」，而不是 Singleton 模式：

- 用 DI 容器管理「這個東西全程只有一個」（生命週期設成 singleton）。
- 但透過「注入」傳進去，而不是全域 `getInstance()`——**依賴明確、好測試（可注入假的）**。

你 csharp Part 4-4 的依賴注入、各種後端框架的 DI，就是這個思路——**「要唯一實例」的需求還在，但用 DI 達成，避開 Singleton 的缺點**。

## 小結

- Singleton = 確保某類別全程「只有一個實例」+ 全域存取點。
- 好處：省資源、全域一致、方便存取。
- **但爭議大**：像偽裝的全域變數、難測試、隱藏依賴。
- 現代多用**依賴注入**來達成「唯一實例」，避開這些缺點。
