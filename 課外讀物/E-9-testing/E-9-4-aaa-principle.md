# [E-9-4] 什麼是好的測試？AAA 原則（Arrange、Act、Assert）

> **目標**：學會用 AAA 原則寫出「清楚、好讀」的測試，以及「好測試」的幾個特質。

## 測試也要寫得好

你學過為什麼要測試（E-9-1）、測試的種類（E-9-2）、單元測試入門（E-9-3）。這篇講「**怎麼寫出『好』的測試**」——因為亂寫的測試，跟亂寫的程式碼一樣難維護。

## AAA：測試的標準結構

好的測試通常遵循 **AAA** 三段式結構，讓人一眼看懂：

```mermaid
graph LR
    A1["Arrange 準備<br/>設定好測試需要的資料/環境"]
    A2["Act 執行<br/>呼叫你要測的那個東西"]
    A3["Assert 斷言<br/>檢查結果是否如預期"]
    A1 --> A2 --> A3
```

| 階段 | 做什麼 |
|------|--------|
| **Arrange（準備）** | 設定測試需要的資料、物件、前置條件 |
| **Act（執行）** | 呼叫「被測試的那個函式/動作」（通常就一行）|
| **Assert（斷言）** | 檢查「結果是不是預期的」 |

## 範例

測一個「計算折扣」的函式，用 AAA 結構：

```javascript
test("VIP 客戶買 1000 元，打 8 折變 800", () => {
  // Arrange（準備）：設定輸入
  const customer = { type: "VIP" };
  const amount = 1000;

  // Act（執行）：呼叫被測的函式
  const result = calculateDiscount(customer, amount);

  // Assert（斷言）：檢查結果
  expect(result).toBe(800);
});
```

三段清清楚楚：準備資料 → 執行 → 檢查。任何人看這個測試，立刻懂「在測什麼、預期什麼」。這比把準備、執行、檢查全攪在一起好讀太多。

## 好測試的特質：FIRST

除了 AAA 結構，好的測試還有幾個特質（業界常用 **FIRST** 來記）：

| 特質 | 意思 |
|------|------|
| **F**ast（快）| 測試要跑得快——慢的測試沒人想跑，就失去價值 |
| **I**solated（獨立）| 每個測試獨立——不依賴別的測試、不依賴執行順序 |
| **R**epeatable（可重複）| 每次跑結果一樣——不依賴「當下時間、隨機、外部狀態」|
| **S**elf-validating（自我驗證）| 測試自己「通過/失敗」很明確，不用人去看輸出判斷 |
| **T**imely（及時）| 在對的時間寫（最好開發時就寫，而非事後）|

## 幾個寫測試的原則

**① 一個測試只測一件事**：一個 test 專注驗證「一個行為」（呼應單一職責，E-7-2）。別在一個 test 裡塞五個斷言測五件事——失敗時不知道是哪個壞了。

**② 測試名字要說清楚「測什麼」**：好的測試名描述「情境 + 預期」，例如 `"VIP 客戶買 1000 打 8 折"`。失敗時，光看名字就知道「什麼壞了」（呼應命名，E-6-2）。

**③ 測「行為」不測「實作細節」**：測「給這個輸入，得到這個輸出」，而不是測「內部用了什麼變數」。否則你重構（改內部、但行為不變）時，測試會亂壞——好的測試應該「重構不壞、行為錯才壞」。

**④ 也要測「邊界」和「錯誤情況」**：別只測「正常情況」。測「空輸入、極端值、會出錯的情況」——bug 常藏在這些邊界（呼應 E-9-1）。

## 為什麼好測試這麼重要

回到 E-9-1 的精神——測試是「讓你敢改程式碼的安全網」。但**只有「好的測試」才是好的安全網**：

- AAA 結構 + 好命名 → 測試好讀、好維護、失敗時好懂。
- FIRST 特質 → 測試快又可靠，大家才願意常跑。
- 測行為不測實作 → 重構時測試不會亂壞，真正保護你。

爛的測試（攪成一團、慢、不穩、測實作細節）反而是負擔——大家不想跑、一改就壞、還要花時間維護。所以「寫測試」和「寫好測試」一樣重要。

## 小結

- **AAA**：測試三段式——Arrange（準備）、Act（執行）、Assert（斷言），讓測試一眼看懂。
- **FIRST** 特質：Fast、Isolated、Repeatable、Self-validating、Timely。
- 原則：一個測試測一件事、名字說清楚、測行為不測實作、測邊界與錯誤。
- 只有「好的測試」才是好的安全網——爛測試是負擔。
